## 零、险种分类知识体系

> 本节是保险行业的领域知识基础，影响后续所有设计决策。基于《中华人民共和国保险法》第95条及银保监会分类标准。

### 0.1 第一层：按保险标的分类

```
保险业务
├── 人身保险（以人的寿命和身体为标的）
│   ├── 人寿保险         → 定期寿险、终身寿险、两全保险、年金保险、万能险、投连险
│   ├── 健康保险         → 医疗保险（百万医疗）、重疾保险（确诊即赔）、护理保险
│   └── 意外伤害保险     → 综合意外险、交通意外险、旅游意外险
└── 财产保险（以财产及其有关利益为标的）
    ├── 车险（机动车保险）→ 交强险、商业车险（车损险、三者险、座位险）
    ├── 企业财产保险     → 财产基本险/综合险/一切险
    ├── 家庭财产保险     → 家财险
    ├── 责任保险         → 公众责任险、雇主责任险、产品责任险
    ├── 信用保证保险     → 履约保证险、贷款保证险
    ├── 货物运输保险     → 货运险（海运/陆运/空运）
    ├── 农业保险         → 种植险、养殖险
    └── 工程保险         → 建筑工程一切险
```

### 0.2 按保障期限

| 类别 | 说明 | 典型险种 |
|------|------|---------|
| 短期险 | 1年以内 | 车险、意外险、医疗险 |
| 长期险 | 多年至终身 | 重疾险、终身寿险、年金险 |

### 0.3 按投保关系

| 类别 | 说明 | 统计影响 |
|------|------|---------|
| 个险 | 个人投保 | 通常1个被保人，偶有多人 |
| 团险 | 企业为员工投保 | 名单制，可多人 |
| 银保 | 银行渠道销售的保险 | 通常为理财型 |

### 0.4 本系统的险种分类（用于统计规则）

实际保单首页的"险种名称"字段是保险公司自定义产品名（如"国寿鑫享至尊年金保险"），需通过关键词匹配识别所属大类：

| 系统分类 | 保单原文关键词（按优先级） | 统计规则 |
|---------|--------------------------|---------|
| **车险** | 车险、机动车、交强险、三者险、车损险、商业车险、驾意 | 财产险规则：只显示投保人，统计1条 |
| **人寿险** | 寿险、人寿、终身、定期、两全、年金、万能、投连、分红 | 人身险规则：显示投保人+被保人 |
| **健康险** | 健康、医疗、重疾、疾病、防癌、护理、医保 | 人身险规则：显示投保人+被保人 |
| **意外险** | 意外、意外伤害、人身意外、交通意外、旅游意外 | 人身险规则：显示投保人+被保人 |
| **财产险** | 财产保险、企财、家财、责任保险、责任险、货运、工程保险、保证保险、信用保险、农业保险 | 财产险规则：只显示投保人，统计1条 |
| **未知** | 无任何关键词命中 | 标记"待确认"，用户手动指定 |

> 判别优先级：**关键词从前往后匹配，谁先命中归谁**。例如"机动车交通事故责任强制保险"会先命中"机动车"→归车险，不会因后续含"责任"被误归为责任险。

## 一、项目背景

金融保险行业在客户信息管理中有大量保单文件。保单首页包含投保人、被保人身份、险种、金额等敏感信息，后续合同页为通用模板，不属于关注范围。

传统扫描工具基于规则提取姓名、手机号、地址等字段，但缺乏对保单业务含义的理解。本工具的目标是：
1. **理解保单内容** — 提取有业务意义的字段（保险公司、险种、投保人、被保人等）
2. **区分险种类型** — 人身险需同时提取投保人和被保人，财产险只需投保人
3. **按个体统计敏感信息条数** — 1个被保人 = 1条有效敏感信息

## 二、整体流程

```
用户上传（PDF/图片）
      ↓
 [文件预处理]
   - PDF → 每页转图片（PyMuPDF）
   - 图片尺寸统一，提高OCR质量
      ↓
 [OCR识别]  ← PaddleOCR（离线，本地CPU运行）
   - 提取所有文字块 + 位置坐标
      ↓
 [页面分类]
   - 判断当前页是否为保单首页
   - 非首页自动跳过
      ↓
 [字段提取]  ← 规则 + 关键词匹配
   - 保险公司 | 险种 | 保单号
   - 投保人 | 被保人 | 受益人
   - 保费 | 交费方式 | 生效日期 | 保险期间 | 销售经理
      ↓
 [险种分类]  ← 新增步骤
   - 根据险种名称中的关键词识别所属大类
   - 输出：车险 / 人寿险 / 健康险 / 意外险 / 财产险 / 未知
      ↓
 [敏感信息统计]
   - 人身险：显示投保人+被保人信息，按被保人去重统计
   - 财产险：只显示投保人信息，固定统计为1条
   - 未知：标记"待确认"，提供修正入口
      ↓
 [结果展示]
   - Web界面：表格列表 + 编辑修正 + 统计汇总
   - 区分显示：人身险显示两列（投保人+被保人），财产险只显示投保人
   - 导出：Excel / JSON
```

## 三、功能模块设计

### 3.1 UploadHandler（上传处理）

| 项目 | 说明 |
|------|------|
| 输入 | 用户上传的文件 |
| 校验 | 文件格式：PDF、JPG、PNG |
| 输出 | 文件保存至临时目录，返回文件列表 |
| 批量 | 支持一次选择多个文件 |

### 3.2 PdfProcessor（PDF转图片）

| 项目 | 说明 |
|------|------|
| 依赖 | PyMuPDF（fitz） |
| 功能 | 将PDF每页转为JPEG图片 |
| 配置 | 输出分辨率 200 DPI，平衡速度与清晰度 |
| 输出 | 图片文件列表（每页1张） |

### 3.3 OcrEngine（OCR识别）

| 项目 | 说明 |
|------|------|
| 引擎 | PaddleOCR（paddlepaddle-cpu 模式） |
| 离线 | 完全离线运行，预下载中文模型 |
| 输入 | 图片文件 |
| 输出 | 文字块列表 `[{text, confidence, bbox}]` |

### 3.4 PageClassifier（页面分类器）

| 项目 | 说明 |
|------|------|
| 功能 | 判断当前页是否为保单首页 |
| 策略 | 关键词匹配（如"保险单"、"保单号"、"投保人"等高频词出现N个以上则判定为首页） |
| 输出 | `True` / `False` |
| 非首页 | 跳过，不影响其他文件 |

### 3.5 FieldExtractor（字段提取器）

| 字段 | 提取策略 |
|------|---------|
| 保险公司 | 常见保险公司名称列表匹配（中国人寿、平安、太平洋、泰康、新华等） |
| 险种 | 关键词"险种"、"产品名称"等后邻文字段提取 |
| 保单号 | 关键字"保单号"后邻文字，或特定编号格式匹配 |
| 投保人 | 关键字"投保人"后邻文字 |
| 被保人 | 关键字"被保险人"、"被保人"后邻文字 |
| 受益人 | 关键字"受益人"后邻文字（可选字段） |
| 保费 | 关键字"保费"、"保险费"后邻数字+单位 |
| 交费方式 | 关键字"交费方式"、"缴费方式"后邻文字 |
| 生效日期 | 关键字"生效日期"、"生效日"后邻日期格式 |
| 保险期间 | 关键字"保险期间"、"保险期限"后邻文字 |
| 销售经理 | 关键字"销售经理"、"业务员"、"代理人"后邻文字 |

提取策略说明：
- 定位关键词 → 根据坐标或文字顺序取邻接字段值
- 支持表格布局（行/列定位）和段落布局
- 置信度过滤：OCR置信度低于 0.7 的字段标记为"待确认"

### 3.6 InsuranceClassifier（险种分类器）← 新增

| 项目 | 说明 |
|------|------|
| 输入 | PolicyFields.policy_type（OCR提取的险种名称原文） |
| 输出 | 枚举值：CAR / LIFE / HEALTH / ACCIDENT / PROPERTY / UNKNOWN |
| 算法 | 关键词优先级匹配（按0.4节分类表，从前到后逐一命中） |
| 位置 | 在FieldExtractor之后，StatisticsEngine之前执行 |

**分类后产生的业务影响：**
- `CAR` / `PROPERTY` → 标记为"财产险类别"，统计时只取投保人
- `LIFE` / `HEALTH` / `ACCIDENT` → 标记为"人身险类别"，统计时取被保人（同时显示投保人信息）
- `UNKNOWN` → 标记为"待确认"，前端提供险种类型选择器由用户修正

### 3.7 StatisticsEngine（统计引擎）

| 项目 | 说明 |
|------|------|
| 统计口径 | **按险种类型区分统计规则** |
| 人身险规则 | 提取被保人姓名 → 去重 → 计数。1个被保人=1条，同时显示对应的投保人信息 |
| 财产险规则 | 提取投保人信息 → 固定统计为1条。不涉及被保人 |
| 未知规则 | 标记为"待确认"，暂不纳入统计。用户手动指定类型后重新计算 |
| 输出 | 敏感信息总条数 + 按险种分组的详细列表 |

**人身险统计逻辑（与之前一致）：**
```
收集所有被保人姓名 → 去重 → 计数
示例：3张人身险保单，2个不同被保人 = 2条敏感信息
显示时附带对应的投保人信息
```

**财产险统计逻辑：**
```
每张保单固定为1条记录，取投保人信息
不按被保人去重（因为财产险的标的不是人）
示例：5张车险保单，同一投保人 = 5条记录（每单1条）
```

### 3.8 ReportViewer（结果展示）

- **技术**：原生 HTML/CSS/JS（无前端框架）
- **功能**：
  - 上传区域（拖拽/点击）
  - 识别进度提示
  - 结果表格展示
  - **区分显示**：人身险显示"投保人"+"被保人"两列；财产险只显示"投保人"列，"被保人"列隐藏或显示"N/A"
  - 险种类型列：显示归类后的大类名称（车险/人寿险/健康险/意外险/财产险）
  - 未知类型行：险种类型列显示"待确认"，并提供下拉选择器修正
  - 单元格双击可编辑修正
  - 敏感信息统计汇总（按险种类型分组展示）
  - 导出 Excel（openpyxl）
  - 导出 JSON

### 3.9 数据模型变更

PolicyFields 新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `insurance_category` | `Optional[str]` | 险种分类结果：car/life/health/accident/property/unknown |
| `display_insured` | `Optional[bool]` | 是否显示被保人列（依据险种判定） |

SensitiveStats 变更：

| 字段 | 说明（变更前 → 变更后） |
|------|------------------------|
| `total_unique_insured` | 仅被保人去重 →改为→ 按险种规则统计后的总条数 |
| `insured_list` | 保留，仅人身险含值 →改为→ 人身险有值，财产险为空列表 |
| `details` | 所有结果 →改为→ 按险种分组的明细 |
| `property_count`（新增） | 财产险保单总条数 |
| `unknown_count`（新增） | 未分类保单数 |

## 四、技术选型

| 层面 | 技术 | 理由 |
|------|------|------|
| 后端语言 | Python 3.11+ | OCR生态好，PaddleOCR原生支持，字段提取逻辑灵活 |
| Web框架 | FastAPI | 轻量、异步、自动生成API文档 |
| OCR引擎 | PaddleOCR (paddlepaddle-cpu) | 离线下中文识别率最高，CPU即可运行 |
| PDF处理 | PyMuPDF (fitz) | 快速将PDF页转图片 |
| 前端 | 原生HTML/CSS/JS | 界面简单，无需前端工程化 |
| 导出 | openpyxl (Excel) | 结构化导出Excel报表 |
| 部署 | `python main.py` 启动，浏览器访问 | 最简单直接 |

> 注：不需要数据库，结果暂存内存，刷新即清空。不包含用户系统。

## 五、异常处理

| 场景 | 处理方式 |
|------|---------|
| 文件格式不支持 | 返回提示，跳过 |
| OCR置信度低于阈值 | 标记"待确认"，人工可修正 |
| 非保单页 | 标记"无法识别"，不影响其他文件 |
| PDF加密/损坏 | 返回错误提示 |
| 批量中部分失败 | 成功的正常展示，失败的单独标注 |
| 重复文件 | 简单覆盖（不做自动去重） |
| **险种无法识别** | 标记"待确认"，前端提供类型选择器由用户手动指定 |
| **财产险无被保人字段** | 被保人列显示"N/A"，不影响统计 |

## 六、非功能性需求

| 指标 | 目标 |
|------|------|
| 离线运行 | 完全离线，不需要联网 |
| 中文识别 | PaddleOCR中文模型，准确率 > 90%（常规印刷保单） |
| 启动时间 | 首次加载模型约 5-15 秒 |
| 每页识别速度 | 约 1-5 秒（取决于图片大小和复杂度） |
| 单机运行 | 仅限本机使用，不支持多用户并发 |
| 险种分类耗时 | 纯文本规则匹配，< 10毫秒，几乎不计 |
