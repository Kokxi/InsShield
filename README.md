# InsShield — 保险行业个人信息监测扫描工具

[![Python 3.10+](https://img.shields.io/badge/python-3.10%2B-blue)](https://www.python.org/)
[![License: MIT](https://img.shields.io/badge/license-MIT-green)](LICENSE)

## 核心区别：找"字段" vs 找"人"

传统数据扫描工具的工作方式是**字段匹配**——在文档中搜索身份证号（18位数字）、手机号（11位数字）、银行卡号等正则模式。它不关心这些信息属于谁，也不关心一份文档中提到的是一个人还是多个人。

InsShield 以**个人信息主体为中心**，将同一人的所有信息聚合为一条"个人记录"，直接输出法规要求的合规口径：

| 维度 | 传统工具（字段中心） | InsShield（人中心） |
|------|-------------------|-------------------|
| 检测目标 | 匹配字符串模式 | 识别一个完整**个人信息主体**所关联的所有信息 |
| 信息组织 | 按类型分类（有多少个身份证、多少个手机号） | 按人聚合（张三→哪些信息，李四→哪些信息） |
| 关系理解 | 不关心信息之间的归属关系 | 识别哪些信息属于同一个人 |
| 多人文档 | 无法区分甲和乙的信息边界 | 能清晰区分各主体的信息边界 |
| 上下文判断 | 只查局部字符模式 | 结合上下文判断身份和关系 |
| **合规口径** | **字段条数**（与法规脱节） | **涉及主体人数**（直接满足报告要求） |

### 实际对比

**传统工具扫描结果：**
> ✅ 找到身份证号 ×1，手机号 ×1，银行卡号 ×1

**InsShield 扫描结果：**
> ✅ 张三个人信息集合：身份证号、手机号、家庭住址、联系方式
> ✅ 李四个人信息集合：身份证号、银行卡号
> ⚠️ 发现一条地址信息无法确定归属，需人工确认

前者告诉你"有哪些敏感字段"，后者告诉你"**有哪些人的信息、各关联了什么、哪些归属不明**"——这是根本性的差异。

## 应用场景

保单文档包含大量客户个人信息，在以下业务场景中尤为突出：

- **理赔协同** — 多家机构联合定损时，涉及多个被保险人的个人信息交叉
- **数据标注** — 外包团队标注保单字段前，需明确每个标注对象的信息边界
- **合规审计** — 监管要求"最小必要"原则，传统工具只看字段数量，无法评估"每个人的信息是否最小"
- **数据发布** — 内部数据集共享或公开研究时，需确保每个个人信息主体都被脱敏

InsShield 通过 **以人为主体的信息聚合 + OCR + 规则引擎**，实现从"检测敏感字段"到"监测个人信息"的升级。

## 法规基础：为何以"人"为单位

传统数据扫描工具按**字段条数**统计（身份证号×1、手机号×1、银行卡号×1 = 3条），而行业监管法规的计算口径是以**个人信息主体**为单位：

| 法规层级 | 法规/规范 | 核心条款 | 计算单位 |
|---------|---------|---------|---------|
| **上位法（框架原则）** | **《中华人民共和国个人信息保护法》** | 第四条以"自然人"定义个人信息，确立了以个人为保护单元的基本定位；但该法是原则性规定，未明确规定以人为单位的计数口径 | 原则性框架，不规定具体口径 |
| | **《中华人民共和国数据安全法》** | 第二十一条要求建立数据分类分级保护制度；该法侧重于数据安全制度框架，未以"人数"为计量单位 | 原则性框架，不规定具体口径 |
| | **《银行保险机构消费者权益保护管理办法》（银保监会令2022年第9号）** | 第四十二条要求建立消费者个人信息保护机制，实行分类分级管理；该办法规定处理规则，未明确规定以人为单位的计数口径 | 原则性框架，不规定具体口径 |
| **下位法及配套规范（明确按主体计数）** | **《个人信息安全规范》（GB/T 35273-2020）** | 第8.2条个人信息安全事件处置中，需按照**涉及个人信息主体数量**判断事件严重等级；附录D明确"个人敏感信息"判定标准 | 个人信息主体 |
| | **《关于办理侵犯公民个人信息刑事案件适用法律若干问题的解释》（法释〔2017〕10号）** | 第十一条：向不同单位或者个人分别出售、提供**同一公民个人信息**的，信息条数**累计计算**；第五条：出售或提供行踪轨迹信息 **50条以上**、健康生理等信息 **500条以上**、一般个人信息 **5000条以上**即构成**"情节严重"** | 每条信息（向每N个接收方分别计数） |
| | **《数据出境安全评估办法》** | 第四条：处理**100万人以上**个人信息向境外提供，或自上年1月1日起累计向境外提供**10万人以上**个人信息（或**1万人以上**敏感个人信息）→ 需申报**数据出境安全评估** | 涉及的个人信息主体人数 |
| | **《个人信息出境认证办法》** | 通过认证方式向境外提供个人信息的适用门槛：自当年1月1日起累计向境外提供**10万人以上、不满100万人**个人信息（不含敏感个人信息），或**不满1万人**敏感个人信息 | 涉及的个人信息主体人数 |

> ⚠️ **刑法的放大效应**：一份保单包含 N 个个人信息主体，每人有 M 项信息条目。若向 K 家合作机构分别提供，则刑法定罪所用的"信息条数" = N × M × K。传统数据扫描工具按字段去重统计（仅 M），无法暴露这个乘法放大风险。InsShield 按主体聚合后，可直观看到"哪些人的信息被提供给了多少家机构"，让合规部门能准确评估刑事风险。

**关键逻辑**：一个人同时有身份证号、手机号、家庭住址、银行卡号、职业信息，从法规角度看，这属于**1个个人信息主体的数据集合**，而非5条独立数据条目。合规报告、安全事件定级、处罚力度，都基于"涉及多少个主体"，而非"涉及多少个字段"。

## 功能特性

- **以人为主体的信息聚合** — 自动将同名、同身份证号等信息关联至同一主体，区分不同人的信息边界
- **多格式文档解析** — 支持 PDF 和 Word（`.docx`）文档，含扫描件
- **OCR 文字识别** — 基于 RapidOCR，自动识别 PDF 中的扫描图片文字
- **PII 智能检测** — 识别身份证号、手机号、银行卡号、地址、姓名等敏感信息，含校验逻辑
- **文档智能分类** — 自动区分身份证、保单、银行流水等文档类型
- **结构化字段提取** — 按页面布局提取关键字段，保留业务语义
- **可视化预览 Web 界面** — 上传 → 分析 → 高亮标注敏感区域 → 脱敏导出，一站式操作
- **统计看板** — 文档处理进度、PII 类型分布、各文档敏感度排名、工作量趋势（按日/周），为脱敏决策提供数据支撑
- **脱敏导出** — 导出脱敏后的文档，安全共享

## 快速开始

### 环境要求

- Python 3.10+
- pip / uv

### 安装

```bash
git clone https://github.com/Kokxi/InsShield.git
cd InsShield
pip install -r requirements.txt
```

### 运行

```bash
uvicorn main:app --reload --port 8000
```

打开浏览器访问 http://localhost:8000

### 运行测试

```bash
python -m pytest tests/ -v
```
